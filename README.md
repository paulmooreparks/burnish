# burnish

A tool for auto-editing LLM output to fit a target writing style, whether that
style belongs to one author or to a whole body of work.

Point it at a corpus of writing; it distills that corpus into a structured
**style profile**, then measures how far any draft sits from that style and helps
revise it closer. The goal is to move style enforcement *out* of the generating
model (where in-context rules are quickly forgotten) into a separate,
deterministic-plus-adversarial checker.

See [DESIGN.md](DESIGN.md) for the architecture and [CARRYOVER.md](CARRYOVER.md)
for current status.

## Use cases

There are two ways to point it, and **both are first-class**:

- **One author's voice.** Distill a single author's corpus and revise new drafts to
  match how *they* write.
- **A house / consensus voice.** Distill a body of work written by **many authors**
  (a knowledge base, a documentation set, a customer-facing corpus) into the common
  voice they share, then revise new output so it reads as if written by *yet another
  author who fits the existing body*. Example: generating customer-facing documents
  for a software system being customized for one customer, so each new document
  blends into the larger documentation set instead of standing out as off-voice.

The house-voice case is, if anything, **more forgiving** than single-author
cloning: you are matching the *envelope* the corpus occupies (its central tendency
plus its spread), not cloning one person exactly. burnish's discriminator asks "did this come from our
body of work?", so "on-target" means "indistinguishable from a document already in
the set."

The one rule for either mode: a corpus must be **single-register** (one genre /
document type). **Many authors is fine, even ideal**; it is many *registers*
averaged together that produces mush. See
[docs/preparing-a-corpus.md](docs/preparing-a-corpus.md).

## Status

The full **deterministic, no-model engine is built** and exercised end to end:

- **distill** a corpus into a profile (features + lexicon + corpus-validated rules)
- **score** a draft: distance-to-style, off-target features, rule violations
- **calibrate** a discriminator (a threshold separating target text from off-style
  decoys) and get an on-target / off-target verdict
- **retrieve** topically relevant target-style exemplars (TF-IDF, no embeddings)
- the **massage loop** (`enforce`/`pkg/api`): lint -> judge -> retrieve ->
  discriminate -> revise, bounded
- an **MCP server** (the primary agentic surface) and a **Claude Code Stop hook**
- a headless **`model/` adapter** (a stdlib-only Anthropic Messages client) and a
  Fielding-style **`serve`** HTTP API, for agent-less callers (.NET sidecar, CI)

The architecture keeps inference in the **caller**: the deterministic stages need
no model, and the revise/judge steps are rendered by the calling agent's LLM (the
agentic path), so the engine bakes no API key. The optional `model/` adapter wires
an LLM API directly, but only as the headless fallback, and `serve` exposes the
massage action only when an API key is configured.

## Build

```
go build ./cmd/burnish
```

Requires Go 1.25+ (the MCP SDK sets the floor). The binary is self-contained.

## Concepts

A **profile** is a YAML artifact distilled from one *register* (genre) of writing:
weighted statistical **features** with target ranges, a **lexicon** of
characteristic (preferred) and author-avoided vocabulary, corpus-validated
**rules**, and an optional calibrated **discriminator**. Distillation is
per-register: feed one distill run a genre-homogeneous corpus (all essays, or all
release notes, not a mix of genres), or the averages turn to mush. The corpus can
be a single author or **many authors writing that one genre** (a house voice); it
is mixing *genres*, not authors, that averages to mush.

A profile is a stylometric fingerprint, i.e. **user data**: profiles are
gitignored and kept local, regenerated by `distill`. burnish imposes **no
universal style rules** (it is a tool for any author's style); invariants are
per-profile and opt-in (see `--avoid`).

**Distance** is the headline number: the weighted mean deviation, in standard
deviations, of the draft's features outside their target ranges (0 = within every
range; lower = more on-style). **Hard violations** are author-configured avoided
terms (e.g. em-dashes, via `--avoid`); they make `score` exit non-zero so it can
gate a pipeline. The **discriminator** (once calibrated) turns the distance into
an on-target / off-target verdict against a learned threshold.

## Usage

### 1. Distill a profile from a corpus

```
burnish distill --corpus DIR --register NAME [--language en] [--id ID] [--out FILE] [--avoid "—,--"] [--base FILE]
```

`--corpus` is a directory; every `.md` and `.txt` under it (recursively) is one
document. `--avoid` lists terms this author avoids (they become hard violations);
`--base` inherits a shared base profile file (cross-register invariants).

> **Corpus quality is everything.** `distill` reads files raw and measures whatever
> it finds, so the profile is only as good as the prose you feed it. Before you
> distill, read [docs/preparing-a-corpus.md](docs/preparing-a-corpus.md): how to keep
> the corpus single-register and authentic to the voice you are targeting (one
> author, or a whole body of work), and how to extract clean prose from HTML and
> front-mattered markdown (the failure mode that quietly poisons a profile).

```
burnish distill --corpus ./corpus/essays --register personal-essay --id me --avoid "—,--"
```
```
distilled 25 documents (15976 words) -> me.profile.yaml
  49 features, 40 preferred lexicon terms, 2 deterministic rules
```

Under five documents it warns that ranges are low-confidence. A profile is plain
YAML: readable, diffable, hand-editable.

### 2. Score a draft against the profile

```
burnish score --profile FILE [DRAFT]      # draft from FILE, or stdin if omitted
```
```
profile: me (register personal-essay)
distance to style: 0.478 stddev (0 = within all target ranges)
discriminator: OFF-TARGET (distance 0.478 vs threshold 0.126)

off-target features (worst first):
  first_person_rate    value=2.1   3.10 stddev out  [soft]
  contraction_rate     value=0     2.40 stddev out  [soft]
  ...

rule violations:
  [soft] Sentences stay at or under 45 words; the corpus rarely runs longer.
      para 2: But this particular sentence keeps going and going...
```

Exit code is non-zero on hard violations, so it composes in scripts:

```
burnish score --profile p.yaml draft.md || echo "off-style, blocked"
```

Add `--json` for a machine-readable object (`distance`, `on_target`, `threshold`,
`features`, `lexical`, `hard_violations`, `rule_violations`).

### 3. Calibrate a discriminator

Turn the distance into an on-target verdict by calibrating a threshold against a
decoy (off-style) corpus:

```
burnish calibrate --target DIR --decoys DIR --register NAME [--avoid "—,--"] [--holdout-every N]
```
```
discriminator (distance-threshold):
  AUC:        0.729  (0.5 = chance, 1.0 = perfect separation)
  threshold:  0.1264  (on-target if distance <= threshold)
  at thresh:  TPR 86%, FPR 30%, accuracy 76%
```

It holds out part of the target corpus and scores it against the decoys, so the
threshold reflects target text the profile was never shown. The decoy class is
conveniently generic LLM prose, the thing the tool exists to move away from.

### 4. Retrieve target-style exemplars

```
burnish retrieve --corpus DIR --query TEXT [-k N] [--min-words N]
```

Returns the corpus passages most topically relevant to the query (TF-IDF cosine),
each an authentic target-style passage, as few-shot for revision. No embedding
model or vector store.

### 5. Use it agentically over MCP

```
burnish mcp
```

Runs an MCP server on stdio (official Go MCP SDK), exposing to any MCP client:

- `distill` (`corpus_dir`, `register`, ...) writes a profile.
- `score` (`profile_path`, `text`) returns the distance, on-target verdict,
  off-target features, hard-violation count, and rule violations.
- `style_review` (`profile_path`, `text`) returns a revision payload: the gap
  report, the discriminator verdict, rule violations, retrieved guidance, and the
  profile's preferred/avoided lexicon and rules. The deterministic discriminator
  gives the on-target verdict; the richer LLM-judged subjective rules are an
  upgrade. The calling agent renders the judgement and revision in a fresh,
  isolated context (never the one that wrote the draft).

```
claude mcp add burnish -- /path/to/burnish mcp
```

### 6. Enforce style on every Claude Code turn (Stop hook)

`burnish hook` is a Claude Code **Stop hook**: it checks the last (top-level)
assistant message for **hard** violations of what *you* configure and blocks the
turn so Claude must revise. The deterministic guarantee instruction-based rules
never had: a rule you set cannot be forgotten. It checks hard violations only (a
chat turn is a different register than a distilled profile) and **fails open** so
it can never wedge a session.

burnish imposes no universal rules, so the hook enforces only what you configure:
`--avoid "—,--"` to block terms, and/or `--profile FILE` (or `$BURNISH_AVOID` /
`$BURNISH_PROFILE`). With nothing configured it enforces nothing. To stop
em-dashes on every turn, add to `~/.claude/settings.json`:

```json
{
  "hooks": {
    "Stop": [
      { "matcher": "", "hooks": [ { "type": "command", "command": "burnish hook --avoid \"—,--\"" } ] }
    ]
  }
}
```

On a violation it returns `{"decision":"block","reason":...}` naming the offending
terms; a clean turn produces no output.

### 7. Serve the engine over HTTP (headless callers)

```
burnish serve --profiles ./profiles [--addr :8080] [--model NAME]
```

For agent-less callers (a .NET sidecar, CI) there is no orchestrating LLM, so
`serve` exposes the engine as a **Fielding-style (HATEOAS) REST API**. It loads
every `*.profile.yaml` under `--profiles` and is hypermedia-driven: start at `/`
and follow `_links` to a profile, then POST a draft to its `assessments` or
`reviews` collection (each returns `201` + a `Location` you can GET back).

The **massage** action needs an LLM. If `ANTHROPIC_API_KEY` is set, `serve` wires
the built-in `model/` adapter (a stdlib-only Anthropic Messages client; `--model`
defaults to Haiku) and the `massages` link appears on every profile. Without a key
that link is simply **absent** from the representation (illegal actions are not
advertised), and a direct POST gets `405` with `"reason":"no_reviser"`.

```
curl -s localhost:8080/profiles/me                      # see the available actions
curl -s -XPOST localhost:8080/profiles/me/reviews -d '{"text":"my draft"}'
```

### As a Go library

`pkg/api` exposes `LoadProfile`, `Check` (deterministic assessment), `BuildBank`,
and `Massage` (the bounded loop, taking your LLM as the `Reviser`) for apps that
wrap their own LLM calls. The `model` package provides a ready-made `Reviser`
(`model.NewClient(key).Reviser()`) for headless callers that want the built-in
adapter instead of their own LLM.

## What gets measured

- **Length & cadence:** sentence-length mean and variance, paragraph length, word
  length, Flesch-Kincaid reading grade, markdown heading/list cadence.
- **Punctuation rates** (per 1000 words): em-dash, semicolon, colon, parens,
  exclamation, question, comma.
- **Lexical class rates:** contractions, hedge words, first/second/third-person.
- **Function-word fingerprint:** per-word frequencies of 30 high-frequency
  function words, the classic authorship-attribution signal. (Sampling-error
  corrected, so short drafts are not dominated by single-token noise.)
- **Lexicon:** distinctive vocabulary mined against a 20k-word general-English
  frequency table (built from ~7.3M words of public-domain prose plus a modern
  sample; non-prose tokens like markup are filtered), plus the author's `--avoid`
  terms. Reproducible via `scripts/fetch-baseline-corpus.sh`.
- **Rules:** corpus-validated structural constraints (e.g. max sentence /
  paragraph length) that catch per-instance violations the aggregate distance
  averages away.

## Known limits

- The modern (Wikipedia) portion of the distinctiveness baseline is a modest
  sample, so some everyday-modern words can still read as mildly distinctive;
  more modern text sharpens it further.
- Deep authorial voice (argument structure, humor, rhythm) is asymptotic: the
  discriminator reports a distance and you drive it down, but expect a plateau.
  Frame it as "measurably close and improving," not a perfect clone. Note this
  plateau is most binding for **single-author** cloning; fitting a **house /
  consensus voice** is more forgiving, because the target is a region (the body of
  work's envelope) rather than one exact point.
- Subjective LLM-judged rules and dense semantic-embedding retrieval are tracked
  upgrades; the deterministic first cuts ship today.

## Languages

The design is language-agnostic: segmentation, the feature set, and the lexicon
baseline are a per-language module selected by a profile's `language` field. The
core engine, profile format, scoring, discriminator, and rule layer are
language-neutral, and the LLM half works across languages with no extra code.
Today only the English (`en`) module is built; `distill --language X` and `score`
refuse a language with no module rather than mis-measure it. The text foundation
is Unicode-clean; correct word segmentation for non-spacing scripts (Chinese,
Japanese, Thai) awaits those modules. See DESIGN.md section 11.
